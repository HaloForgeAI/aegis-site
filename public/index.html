Aegis plans first, then acts through explicit tools.

The control plane runs in Docker. Anything that needs your local files, terminal, browser, desktop tools, or MCP stdio servers routes through Aegis Local Gateway on your machine. That split keeps the cloud brain durable while preserving the tools that only your host can safely use.

Fast path

The public quickstart target

The source repo can remain private. Public onboarding needs two public artifacts: the GHCR server image and signed CLI releases for macOS arm64 and Windows x64.

1

Start the control plane

docker pull ghcr.io/haloforgeai/aegis:v0.1.1

Runs the Aegis server stack as a Linux container on Docker Desktop or a Linux host. Public pull requires the GHCR package visibility gate in the launch checklist.

2

Install the local gateway

aegis onboarding doctor
aegis worker tools --no-exec

The gateway is the host edge: filesystem, terminal, browser, and local MCP execution stay on the user's machine.

3

Connect daily channels

aegis status
aegis diagnostics deployment-readiness

Telegram, Discord, Feishu, WeCom, HaloForge, Codex, and Claude Code become surfaces for the same Aegis brain.

Public launch readiness

What must be public before the one-line install.

These are operational gates, not product dreams. The homepage should only show a command as generally available after it passes anonymously from a clean machine.

  • GHCR image visibility: set `ghcr.io/haloforgeai/aegis` to Public so anonymous `docker pull` works even while `HaloForgeAI/Aegis` remains private.
  • Public CLI assets: mirror macOS arm64 and Windows x64 `aegis` downloads into a public release surface.
  • Public installer: host a small bootstrap script that downloads the public compose file, starts the image, installs the CLI, and registers the local gateway.
  • Cloudflare Pages domain: serve this site from `aegis.haloforge.dev` with the `aegis-site` Pages project.
